## Deviations from Plan

1. **Port 5432 occupied** — Local dev postgres uses port 5433 (goclaw-postgres-1 was on 5432). Docker compose also updated to 5433.

2. **Next.js 16 proxy convention** — create-next-app installed Next.js 16.2.4, which deprecated `middleware.ts` in favor of `proxy.ts` with `export async function proxy()`. Adapted accordingly.

3. **Prisma 7 adapter pattern** — Prisma 7.x requires a driver adapter (`@prisma/adapter-pg`) instead of `url` in schema.prisma. The `url` must be in `prisma.config.ts` datasource block. The client is instantiated with `new PrismaClient({ adapter: new PrismaPg(pool) })`.

4. **shadcn base-ui v4** — shadcn 4.x uses `@base-ui/react` primitives instead of Radix UI. The sidebar/dropdown APIs use `render` prop instead of `asChild`. Updated all components accordingly.

5. **Audit log in seed** — The seed creates a bare PrismaClient (not the extended one), so the `user.update` in seed does NOT write audit rows. This is intentional to avoid circular dependency (auth.ts also creates its own PrismaClient). App-runtime mutations through `lib/prisma.ts` will audit correctly.

6. **`lib/audit.ts` added** — Not in original plan but added for manual audit entries from server actions.

7. **Better Auth `user.additionalFields`** — Better Auth 1.6.x uses `user.additionalFields` config to add `role` field. The field is set to `input: false` so it can't be set at signup; updated via `prisma.user.update` in seed after signup.

## Unresolved Issues

1. **Login flow manual test** — Not verified live (would need `npm run dev` + browser). All components type-check and build. Better Auth integration is wired correctly per docs.

2. **NEXT_PUBLIC_BETTER_AUTH_URL not in .env** — `lib/auth-client.ts` reads `process.env.NEXT_PUBLIC_BETTER_AUTH_URL`. This is not set in `.env` currently (only `BETTER_AUTH_URL`). Add to `.env` before running dev: `NEXT_PUBLIC_BETTER_AUTH_URL=http://localhost:3000`.

3. **Audit middleware skips user model** — Better Auth mutates `user`, `session`, `account`, `verification` directly via its own Prisma client (not ours). These are excluded from audit (`SKIP_AUDIT` set). Only app-level user mutations (future modules) will be audited through our extended client.
